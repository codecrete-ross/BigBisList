# Agent Governance

This is the canonical guidance for AI agents and automation working in this
repository. The files here are internal development material and are excluded
from CurseForge packaging.

## Project Invariants

- Big BiS List targets WoW TBC Anniversary only. Keep `BigBiSList.toc` on the
  Anniversary interface line unless a release task explicitly changes it.
- The addon uses a root addon layout for CurseForge packaging. The `.toc` and
  runtime Lua files live at repository root by design.
- Public addon identity is stable: folder `BigBiSList`, saved variable
  `BigBiSListDB`, globals `BigBiSList` and `BigBiSListData`, slash commands
  `/bigbis`, `/bbl`, and `/bbltest`.
- Do not add user-facing documentation or UI text unless the task explicitly
  asks for it.

## Sources Of Truth

- `data/canonical/*.json` is the source for generated addon data.
- `Data.lua` is generated by `tools/generate_lua.py`; never hand-edit it.
- `data/raw/wowhead/*` contains normalized source snapshots used as evidence
  for imports and audits. Full HTML cache files are not committed.
- `data/canonical/scrape_manifest.json` registers source coverage and source
  policy.
- `data/canonical/overrides.json` records reviewed manual corrections.
- `vendor/reference/BIS-TBC-1.15` is a parity and provenance reference, not the
  addon identity.

## Data Workflow

1. Register or update source coverage in `data/canonical/scrape_manifest.json`.
2. Fetch or reprocess Wowhead snapshots only when the task explicitly calls for
   it or source data is intentionally being refreshed.
3. Import snapshots into canonical JSON with `tools/scrape_wowhead.py import`.
4. Use `data/canonical/overrides.json` for manual corrections that cannot be
   derived from source snapshots.
5. Regenerate `Data.lua` with `python tools/generate_lua.py`.
6. Run validation and generation checks before finishing.

Do not use live network scraping for routine code changes. Prefer existing
committed snapshots unless a data-refresh task explicitly requires new fetches.

## Lua Workflow

- `Config.lua` owns addon metadata, saved-variable defaults, and migrations.
- `DataIndex.lua` owns indexing, derived rows, filtering, source summaries,
  ownership helpers, and planner/tooltip data access.
- `Widgets.lua` owns reusable UI primitives.
- `UI.lua` owns the main frame, tabs, filters, rows, settings, and bank scans.
- `Tooltip.lua` owns tooltip hooks and tooltip rendering.
- `Minimap.lua` owns minimap button state and interactions.
- `Core.lua` owns addon load, slash commands, smoke/status commands, and startup
  initialization.

Keep new runtime state under `BigBiSList` or saved variables already owned by
the addon. Avoid new globals. Keep Lua compatible with the target WoW client and
guard optional game APIs with capability checks or `pcall` where the surrounding
code already does so.

## Agent Safety

- Check `git status --short --branch` before editing and preserve unrelated
  dirty worktree changes.
- Do not revert, reformat, regenerate, or stage files you did not intentionally
  change for the task.
- Do not run destructive git or filesystem commands unless the user explicitly
  asks for them.
- Keep generated output deterministic. If canonical data changes, regenerate
  `Data.lua` and include both in the same change.
- Keep internal docs internal. `.pkgmeta` must ignore `AGENTS.md`, `CLAUDE.md`,
  and `docs`.

## Required Checks

- General: `python -m unittest discover -s tests`
- Canonical data: `python tools/validate_data.py --json`
- Generated Lua: `python tools/generate_lua.py --check`
- Manifest coverage: `python tools/scrape_wowhead.py coverage --summary --strict`

For data refreshes, also run the relevant snapshot and requirements audits for
the changed data family before release.
